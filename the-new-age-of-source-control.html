The Architecture for Source Control After Git

Thaddeus

Source control is the wrong category. Thaddeus is the one that replaces it — a live, permissioned, agent-native code substrate for an age of secrets, coordinated security, private-by-default work, and a million agents writing code in parallel.

Git is not the next thing to improve. It is the last generation's local maximum — the way Git itself made SVN the generation before. Git won by being so much better than what came before that we mistook its assumptions for laws of physics: that a project is one public pile of files you clone, that the repository is the unit of permission, that history is a chain of human commits, that there is always a real disk underneath. Every one of those is a Linux-kernel assumption from 2005, and every one is now a load-bearing wall standing in the way of how we actually build. Thaddeus does not add features to that world. It names a different one. Code is not a pile of text files; it is a live, structured, encrypted graph you query. A change is not a commit a human curates; it is an operation a principal — human or agent — signs. Review is not one person reading a diff; it is policy, proof, and reputation. This brief diagnoses why Git and GitHub are the wrong primitives — concretely, problem by problem — and then specifies the system that subsumes both.

Orientation

00Why now: the third generation

Source control moves in generations, and we are standing at the seam between the second and the third.

Generation one was centralized: SVN, Perforce, the world of Phabricator and managed source servers. Generation two began when Git proved good enough to start a new lineage, and a family of platforms grew on top of it — GitHub, then GitLab, Bitbucket, Gitea, Forgejo/Codeberg. The crucial thing about Gen 2 is that none of its members are trying to be different. They are the same idea executed better, cheaper, or more open. GitLab is an open, enterprise-priced GitHub with worse UX. Bitbucket is a cheaper GitHub that integrates with Jira. Forgejo is a fast, genuinely free, self-hostable GitHub clone — and a lovely one — but still a clone. Each is to GitHub roughly what Atom was to Sublime: a sideways step, not a leap.

Generation three has not been built. Two forces make it both necessary and, for the first time, affordable. The first is agents: software that reads every public patch the instant it lands, generates hundreds or thousands of lines per session, and wants its own working copy of the code — many of them at once, in parallel, in stripped-down sandboxes. Tooling designed for one human typing in one place is a poor fit for that world. The second is that GitHub itself is unfurling — reverting merges, posting downtime measured in days rather than minutes, and visibly straining under agent-driven traffic. The platform that became everyone's home no longer feels like stable infrastructure.

Rebuilding something as large as Git and GitHub used to be economically absurd: too expensive to build, too unlikely to win users away from the incumbent. Agents have collapsed that cost. The constraint that kept Gen 3 theoretical has lifted — which is exactly why this is worth specifying in full.

A Gen 3 system is not a faster Git. It changes what the words mean.

the questions a ground-up rethink has to answer out loud

The mistake every "GitHub alternative" makes is to keep Git's nouns and polish its verbs — a better diff viewer, a faster clone, a nicer CLI. That is a Gen-2 move: GitLab, Bitbucket, Forgejo are all the same idea executed better. A Gen-3 system has to put the nouns themselves on trial. When the primary authors are a million agents and not one human, the honest design questions are not "how do we make commits nicer?" They are:

  • Is the file still the unit? Agents do not think in files; they think in symbols, functions, types, and call graphs. Versioning text means every agent re-parses, every conflict is a line conflict, and meaning is bolted on after the fact.
  • Is the repository still a thing? If agents spawn millions of working copies and edit continuously, "a repo you clone" stops describing reality. The unit may be a capability-scoped view over a shared graph, not a physical pile you copy.
  • Is the branch still a thing? If history records itself and operations converge continuously, branches and the merge ritual mostly dissolve into ongoing convergence plus lightweight personal views.
  • Is the human reviewer still the gate? Nobody reads an 11,000-line agent PR. Review at machine scale becomes policy, proof, and reputation — humans set the rules and adjudicate exceptions; they do not read every line.
  • Are agents users, or authors? The whole reason Gen 3 is possible is that agents write most of the code. A system that treats them as heavy clients of human tooling — rather than as first-class principals with identity, reputation, budgets, and accountability — has missed the point of its own moment.

This brief takes a side on every one of those. The answers are the design. And they add up to a claim bigger than "a better Git": that the category called source control — a versioned pile of files you clone — is itself the wrong abstraction, and the thing that replaces it is not version control with features added but a different kind of system that happens to subsume what Git and GitHub did.

Part I

01The diagnosis

The problem has two layers. The primitive — Git — encodes the wrong assumptions. The platform — GitHub — is failing at the scale and shape of work we now do. Each complaint below carries an ID so the design in Parts III–V can answer it by name.

The primitive is wrong — Git

P1–P8, P14 · the abstraction itself

P1

Permission is the repository, not the content

Git has no concept of permission below the level of the whole repository. Access is all-or-nothing: clone the repo and you hold every byte in it, including every byte it ever contained. The canonical wedge is the question everyone is trained not to ask — why can't you commit a .env file? The honest answer isn't security; it's "that's just how Git works." Once a secret is in the tree, everyone with access has it; once it's in history, it's there forever. Open-source the project later and the secrets ship with it. Fire someone and they keep the keys — you probably never handed them the file properly anyway. Lend the repo to another team and they get secrets they never needed. An entire industry of secret managers exists to paper over this, and every one of them ultimately resolves to "a random file on your computer." That industry is the symptom; the missing primitive is the disease.

P2

Visibility is a repo flag, not a per-change option

There is no private file inside a shared repo, no private branch, no pull request that stays hidden until it merges, no way to delay when a merge becomes visible to the rest of the team. Public-versus-private is a property of the whole repository instead of a property of a change. Treating it as a repo-level setting rather than a change-level option is, bluntly, insane — and it is wired so deeply into Git that none of the above can be bolted on.

P3

Public-on-merge manufactures zero-days

Because everything is visible the instant it lands, anyone — increasingly, agents — can read every patch as it appears. For a project like Linux, people now run agents over every commit looking for anything that smells like a security fix and derive the exploit before the fix is even announced. The patch is the disclosure. So fixes sit unpublished. We are, in the sharpest line of the argument, "in the middle of a security crisis and bickering over where to store files." There is no way in Git to merge a fix, cut a release, ship it to vulnerable downstreams, and patch the world before the code becomes public.

P4

"Open source" shouldn't mean 100% public, 100% of the time

A great deal of software would be open if developers weren't forced into total, permanent transparency. You should be able to hide in-flight work that may never ship — half of it gets abandoned, and no one wants their dead ends on display. You should be able to keep some sub-packages of a monorepo private without shattering the project into many repositories just to draw a visibility boundary. Having to shape your work around what you want to share — instead of using the tool to shape what gets shared — is backwards. "Open source" should be able to mean open enough.

P5

Commits are a mediocre base unit; branches are worse

Commits are a reasonable unit but a poor fit for how we build now, and branches compound the problem by forcing you to hold the shape of your history in your head at all times. You are made to decide how to record history at the exact moment you are trying to think about the code. Jujutsu (jj) already shows the way out: the working copy is tracked and snapshotted automatically, history captures itself as a side effect of any command, and you simply edit — how it's recorded is a later concern. That it feels so much better is proof of how much attention Git's model quietly taxes.

P6

Worktrees are atrocious

Worktrees are Git's answer to "I need two checkouts at once" — exactly what parallel agents demand — and they are miserable. You arrange copy-on-write by hand though the filesystem could do it for free. You cannot check out the same branch in two places: in one real incident an agent checked out main in a worktree and held it hostage, so main could no longer be checked out in the main directory. Keeping worktrees synced against main is constant manual toil. As the substrate for many agents each needing their own working copy, this stops being an annoyance and becomes a hard blocker.

P7

It demands a real OS and a real filesystem

Interfacing with source control through a CLI against real files on a real kernel looks outdated next to tools like just-bash — a JavaScript/TypeScript layer that emulates bash so an agent can run entirely in memory, with no real Linux kernel and no real filesystem, none the wiser. Reading and writing files should be plain API calls. It is far cheaper to clone things around in memory than to physically move piles of files around a disk — and agents increasingly live in exactly the lightweight sandboxes where standing up a full OS just to talk to source control is pure waste.

P8

The filesystem itself is a platform-specific performance trap

Even with elite hardware, the disk is a minefield. A widely-cited benchmark recreates a fully-cached pnpm install — pure file creation, no network — and measures how long it takes to write the files:

Machine & OS Same cached install
Older mid-range AMD, ample RAM, ordinary WD SSD — Ubuntu ~6.8 s
M4 + Apple's fancy SSD — macOS ~31 s
M1 Ultra — macOS up to ~140 s

Pure file creation from cache, no network access. APFS falls apart on many small files (fsync is the suspected culprit) — and spinning up small agent environments is precisely the small-file workload it handles worst. A task an Ubuntu box finishes in seconds can take an M1 Ultra over two minutes. In-memory representations sidestep the entire rat's nest.

P14

The file is the wrong unit — Git versions text, not meaning

Git's unit of work, visibility, and conflict is the file, and its unit of change is the line. That made sense when a human typed text into a buffer. It does not survive contact with how code is actually written now. A rename is a delete-plus-add that Git only guesses at; a signature change is a wall of unrelated-looking line edits; a formatter run is a "diff" that touched nothing real. Two agents editing the same function on different lines "conflict" over whitespace, while two agents that genuinely break each other's contracts merge clean. The thing the codebase actually is — symbols, definitions, references, types, a call graph — exists nowhere in the model; it is re-derived, expensively, by every tool and every agent on every read. Versioning text and bolting meaning on afterward is backwards. Meaning is the artifact; text is one rendering of it.

Root cause

Why is all of this missing? Because Git was built for Linux-kernel development: one open project, fully public, where the repository is the unit that carries permission, where a real machine with a real filesystem is always present, and where one human reads patches in plain text. Those assumptions sit so deep in the architecture that the missing capabilities can't be patched in later — they would require rethinking the primitive itself. P1–P8 and P14 are not bugs. They are the foundation showing through. The honest conclusion is the one the industry keeps flinching from: you do not fix a foundation by renovating the house on top of it. You pour a new one.

The platform is failing — GitHub, and every alternative to it

P9–P13, P15–P21 · the layer built on top, and the graveyard beside it

P9

Reliability and scale collapse under agent throughput

This is not abstract "downtime measured in days" — it has a shape, and the shape is data loss. A merge-queue regression made roughly 2,847 already-merged PRs appear reverted: a split-brain where a commit that was deployed to production no longer existed in the source of truth, so re-merging it re-ran database migrations that had already run. The official status page called it a 99.5% success rate; the leadership line was that this was only 0.07% of operations — a number reached by widening the measurement window to a full day so the spike would average out. Separately, a webhooks outage silently dropped deploy hooks, so a merged production hotfix simply never shipped for two hours and no alarm fired. And the uptime itself is theater: the official chart shows 99.5%, while the community-run status page clocks the real figure at 86.75% — not even a meme nine.

Underneath, it is Ruby that horizontally scales barely, now buckling under a surge in commits, pull requests, and new repositories — on the order of twenty million new repos — driven by agents. It was built for orders of magnitude less traffic than agents now generate. This is not a law of nature: Pierre's code.store absorbed roughly 9 million repos in thirty days and peaked at about 15,000 repos per minute for three hours straight with no downtime, because it was built for that load. The answer is not a better status page. It is an operation log where a "revert" cannot vanish a deployed commit, on a substrate engineered for the throughput from day one.

P10

Discoverability and navigation are broken — and the "alternatives" are worse

The data model doesn't surface what people actually need, and GitLab — the alternative most people reach for — makes the case better than GitHub ever could. Land on the GitLab project itself and the honest first reaction is "what am I looking at?": the README is 75% of the way down the page, and every number in the sidebar — branches, tags, environments, releases — carries the same visual weight, so the thing everyone actually came for is buried among things nobody clicks. Try to find a commit from last year: infinite scroll, no date filter, and your best option is to clone the repo and search locally. Release pages show a meaningless "88% complete" on a historical release that already shipped, and bury the date at the very bottom, beneath the entire changelog — the useful data last, the useless data first. You can navigate to a commit's parent but there is no next-commit button to its child, even on main, so you cannot step forward to "what broke right after this tag." And the loading is actively hostile: clicking into a repo and hitting back makes the list disappear, because the content was hidden under a second loading layer that never re-fires its request. READMEs sit below the file list on essentially every one of these platforms. You refresh constantly for fear of losing state on navigation. The information architecture is an afterthought layered on a model that was never designed to answer these questions.

P11

Onboarding and the CLI are a generational embarrassment

Creating a repository with the GitHub CLI is nine separate prompts with four loading blockers — and a six-year-old bug that crashes the entire flow if you hit an escape sequence while editing your input. The generational alternative is a single line of code: 

// GitHub CLI: 9 prompts, 4 loading stalls, a 6-year-old crash bug
$ gh repo create

// code.store
const store = new GitStorage()
const repo = await store.createRepo()

Forgejo simply lets you push to a remote and creates the repo on the fly, so an agent can just push. Friction this basic is fatal when the primary author is an agent doing it thousands of times a day.

P12

History records what changed, not why

Git preserves what changed and nothing about why. When an agent generates hundreds or thousands of lines per session, that lost context compounds: agents retrace steps and duplicate reasoning because the intent behind a change was never durably recorded. The race to bolt this back on is already crowded — and tellingly, the people who ran the old world are the ones running from it. Entire — founded by GitHub's former CEO, off a $60M seed — is building a context layer for exactly this. Zed's Delta DB records and synchronizes changes as operations via CRDTs, capturing the how-and-why in real time rather than as after-the-fact snapshots. Cursor shipped Origin aimed at the same gap from the editor side. Three serious, well-funded attempts to graft "why" onto Git — none of them a coincidence, all of them prosthetics for a primitive that records the wrong thing. The "why" is missing from the primitive, and that gap is widening fast.

P13

Community and identity are the crown jewel — and the real casualty

The thing GitHub does best isn't Git hosting; it's being the home. Click a username and you see twenty years of who someone is and what they've built; an npm package leads to its source, its author, its issues — all in one place, the way email and phone numbers became defaults. This is also why the exodus is not hypothetical: the people leaving are the people who built the place. Mitchell Hashimoto and the creator of Ghosty are already moving code off GitHub, and once the maintainers whose profiles were the graph scatter to Forgejo, GitLab, and self-hosted instances, that single shared identity graph fractures — and you cannot tool your way back to it. You can no longer just ask someone for their GitHub profile to know who they are, because their best work is now somewhere else. It is a graduation moment: everyone goes their own way, the twenty-year shared history ends, and no aggregator can rebuild the gravity. Any Gen 3 system that ignores identity and reputation doesn't merely miss a feature — it forfeits the one thing that made the old home worth living in.

P15

Human review collapses — the diff is the wrong surface at agent scale

"No one's reading a PR like this. 11,000 lines added." The review model assumes one human reads the changed lines, in roughly the order the tool lists them — which is, absurdly, alphabetical by path. That was strained when people wrote the code. It is broken now that agents write most of it and open enormous changes constantly. Tools are already routing around the gap: CodeRabbit re-stacks a PR into layers that surface what matters; Graphite rebuilt the entire review experience on mirrored infrastructure because GitHub's was too slow and too flat. But these are prosthetics on a primitive that was never meant to carry this load. Reading raw lines does not scale to a million authors, and no amount of better diff rendering changes that the diff is the wrong thing to be reading.

P16

Agents are second-class — no identity, accountability, or economy for the primary authors

The system's busiest authors are now agents, and the platform has no concept of them. There is no agent identity distinct from the human who runs it, so a change is signed by a person who never saw it. There is no agent reputation — no record that this agent's changes pass review 98% of the time and that one introduced three regressions last week. There are no scoped capabilities or budgets — an agent can touch anything the human can, open unlimited changes, and spend unbounded compute. When an agent is compromised — runs a malicious npx command from a skill.md, or ships the is-odd-with-a-zero package — there is no key to revoke and no blast radius to quarantine. And there is no economy: no way to pay a verified third party 50 cents to audit a change and attach the attestation, the way Socket already detects npm exploits before npm does. The primary authors of the world's code are unmodeled, unaccountable, and unpriced.

P17

The codebase is inert — you cannot query it, subscribe to it, or run policy over it

A repository is a dead pile of bytes you scroll. You cannot ask it "every function that still calls this deprecated API," "everything an untrusted agent touched in the last hour," or "every place this secret-capability is required" — questions that are trivially a query against a live model and impossible against a tree of text. You cannot subscribe to a semantic event ("tell me when anyone changes the signature of this public function") — only to coarse webhooks on pushes. You cannot make a policy stand and run continuously ("no untrusted agent may modify auth code"); policy is a CI script that runs late, after the change already exists. GitHub's broken navigation — infinite scroll, no date filter, a parent-commit link but no child, the date buried under the changelog — is the visible symptom. The disease is that the codebase is not a database, so it cannot answer questions about itself.

P18

Every alternative is a Gen-2 clone — none rethinks the primitive

The honest survey of "where do I go instead?" ends nowhere, because every option is the same generation executed sideways. GitLab is an open, enterprise-priced GitHub with worse UX — "similar but just worse in every single way except uptime," doing nearly a billion dollars a year licensing to enterprises on the strength of being cheaper, not better. And it cannot be fixed by anyone: cloning the GitLab repo takes about five minutes just to start enumerating ~759,000 Git objects, then resolves to 12.78 million lines of code — 3.8 million of Ruby, 600K of Vue components (still Vue 2, not 3), across 528,000 commits. You do not vibe-code your way out of that. The mass is the verdict: this is an old, heavy project, and its weight guarantees the wrongness is permanent. Bitbucket is a cheaper GitHub that exists to integrate with Jira. Forgejo/Codeberg is a fast, genuinely free, lovingly built clone — and the right Gen-2 answer — but its own FAQ describes it as a GitHub-shaped forge. Behind them is a graveyard: sourcehut ("not for us"), CodeCommit (dead), Google Code (dead — Google once believed Git was the wrong solution, then changed its mind and adopted it anyway), SourceForge, Phabricator (dead), and Pierre, which tried to rethink GitHub from first principles, found nobody wanted yet another alternative, and paused the product to build primitives instead. The pattern is total: in twenty years, no one built a durable thing that is actually different. None of them rethinks the permission primitive (P1) or the unit of change (P14); they keep Git's nouns and polish its verbs. That is the definition of a generation that has run out of road.

P19

The "cheap alternative" is lock-in, not a rethink

Bitbucket is the tell. Atlassian doesn't sell it as a better source-control system; it sells "up to 10x savings" and "best-in-class Jira integration." The savings are manufactured: the headline 10x is computed against a GitHub plan deliberately loaded with the $21/user tier plus a $50/user security add-on plus premium support, stacking optional line items to make the comparison look as lopsided as possible. Strip the theatre and the actual value proposition is a single sentence — it integrates with the other Atlassian things you already bought. The page mentions Jira more than it makes a case for itself; with 15 million developers, Atlassian could surface only three customer quotes, one of them an analytics vendor. Choosing it is not choosing a different model of source control; it is deepening a vendor relationship and calling it a migration. A "cheaper GitHub welded to a proprietary issue tracker" is the opposite of the open, portable substrate the moment demands.

P20

Governance is a feature — the platform can be rug-pulled out from under you

Who controls the platform determines whether your home stays your home. Gitea was the popular open self-hostable forge until a for-profit owner took it over and steered it private — stripping "open" from the pitch and putting it behind a license — and the community, feeling rug-pulled, forked it into Forgejo (October 2022) under the democratic nonprofit Codeberg e.V. The difference in governance compounds into real divergence: Forgejo localizes through open Weblate where Gitea uses closed Crowdin, ships security fixes faster, dog-foods its own CI, and — when its hosted actions went down to prep for the copyfile Linux CVE — posted minute-by-minute status updates on a public Mastodon account. That is the kind of transparency you would "kill for from GitHub." Because the other side of the governance coin is a vacuum: GitHub has had no CEO since its last one left to found a competitor, now reports up through a Microsoft EVP, and reportedly dissolved the wall between product and engineering — so the platform everyone's code lives on is steered by an owner with its own priorities and no one clearly accountable for it. Source control has no native answer to "what happens when the steward's incentives change," and that is itself a defect of the platform layer.

P21

The home for code is itself a supply-chain attack surface

The platform that hosts the world's source is also one of its largest unguarded attack surfaces, and it has already been breached at both ends. At the infrastructure end, a Whiz researcher achieved remote code execution via git push -o — an unsanitized push option broke out of its handler and ran code on the server — exposing millions of private repositories from a single crafted header. At the ecosystem end, GitHub owns npm and still let tanstack be squatted, shipping an environment-file exfiltrator under a name developers trusted; the broader pattern includes the is-odd-with-a-zero class of packages and malicious npx commands smuggled through a skill.md. When the substrate stores every secret as plaintext in the tree (P1) and has no concept of an agent identity to revoke (P16), a breach is catastrophic by construction: the attacker gets cleartext, and there is no key to rotate and no blast radius to quarantine. Meanwhile a third party — Socket — detects npm exploits before npm itself does, which is the proof that the platform could be a security boundary and simply isn't. Source control that is not also a security boundary is, in an age of agent-driven supply-chain attacks, negligent by design.

Part II

02The floor everything is built on

A replacement that fixes Git's primitive but abandons what GitHub got right is not an upgrade — and one that wins on features while losing on speed, safety, or cost is not one either. Before designing anything, set the floor everything stands on: the capabilities a successor must not regress on, and the measurable standards every primitive is built to hold.

What we must not lose

capabilities a successor cannot regress on

  • A server-backed remote with real collaboration. A place that holds the code and lets many people — and agents — work on it at once.
  • The social contract of proposing and integrating change. Not the pull request — the contract underneath it: that a change can be proposed, scrutinized, and accepted on the merits. Thaddeus keeps the contract and discards the ritual. Review stops being one human reading a diff and becomes policy, proof, and reputation (Pillar 10); the 11,000-line PR no one reads is not a thing to render better but a thing to retire.
  • Community and discoverable identity. Profiles, contribution history, feeds, stars — the ability to click a name and understand a person. This is the hardest thing to keep and the most valuable.
  • CI/CD. First-class continuous integration and delivery, not an afterthought or a bolt-on.
  • Operational stability + an open, self-hostable core under accountable governance. The Forgejo/Codeberg lesson, learned the hard way: when Gitea was taken private by a for-profit owner, the community forked it (Forgejo, under the nonprofit Codeberg e.V.) because who governs the platform turned out to matter as much as the code. Keep all three — a genuinely free, self-hostable engine; a dependable hosted option; and transparent, non-capturable stewardship (their Mastodon-grade incident updates are the bar) so the home cannot be rug-pulled out from under the people who built it (P19, P20, resolved in Pillar 07).

What everything is measured against

safety · performance · resource cost — each with a number attached

A foundation you cannot measure is a foundation you cannot trust. Git and GitHub's failures here are not matters of taste — they are numbers: a cold checkout an M1 Ultra needs ~140 seconds for, downtime counted in days, one physical worktree per agent. So every primitive in this brief — substrate and commons alike — is held to three measurable standards, and each carries a target. The rule mirrors the traceability matrix: if it can't be measured, it can't be claimed. These are acceptance criteria, not aspirations — the bar a build either clears or it doesn't.

Part III

03The substrate

This is not "a better Git." It is the layer that replaces what Git was for. Six primitives — content-addressed and encrypted, time-varying in visibility, operation-native, provenance-bearing, OS-free, and structured as meaning rather than text — compose into one substrate. The unifying move: take every burden Git pushes onto the author — managing secrets, managing history, managing copies, managing the environment, managing what the code even means — and absorb it into the primitive itself. (Pillars are numbered as stable identifiers, not by position; 08 is a substrate primitive added here, 06–07 and 09–11 live in the commons.)

The decision that comes first — Gen 3 leaves Git

Every honest Gen-3 design has to answer one question before anything else: do you keep Git's content-addressed object model, or break clean? This brief breaks clean, and says so up front rather than burying it as a risk. The source of truth is an operation log — signed, CRDT-ordered operations that record change as it happens. Content-addressed, immutable snapshots still exist, but as a derived, cached projection of that log for fast reads, verification, and migration — not as the primitive. This inverts Git: in Git the snapshot is the truth and history is a story you tell about snapshots; in Thaddeus the operations are the truth and snapshots are a view. Gen 1→2 was SVN→Git, a clean break in the primitive; Gen 2→3 is the same kind of break, and pretending otherwise is how you end up with a faster Git instead of its successor.

Compatibility is then a bridge, not a constraint: Thaddeus can emit a Git history (every operation folds down to commits; every object to a blob; every named view to a branch) so git clone and git push work on day one. But Git cannot emit a Thaddeus history — operations, capabilities, provenance, and structure have nowhere to live in it. The arrow points one way on purpose. Git becomes an export format the way SVN dumps did, not the foundation the new system is forced to inherit.

01

Encrypted objects with per-object capabilities

Capabilities and encryption attach to the objects and operations themselves — not to a repository, and not as "Git's object model with extras." An object (a content-addressed snapshot of a value, derived from the operation log) carries a policy: who may see it, who may decrypt it, and from when. Visibility becomes a property of (object × identity × time). There is no repository-level switch for it to override, because there is no repository in the Git sense — only scoped views over a shared graph.

Mechanics — Objects are stored encrypted (envelope encryption). A capability is the object's content key, wrapped for a specific identity's public key — and identities include agents, not only humans (Pillar 09). To grant access is to wrap the key for that identity; to revoke is to rotate the key and re-wrap only for the remaining grantees. A secret committed to the tree is just an object whose decryption capability was granted to two principals: history holds ciphertext, never plaintext, so "it's in history forever" stops meaning "everyone has it forever." Firing someone — or quarantining a compromised agent — becomes a key rotation, not a doomed scramble. This is also the structural answer to the platform supply-chain breach (P21): when a git push -o RCE or a hijacked dependency reaches the store, what it can read is ciphertext gated by capability, not a tree of plaintext secrets — the blast radius is bounded by what keys the attacker holds, not by what bytes the repo contains. And it is the reason no Gen-2 clone counts as a successor (P18): GitLab, Bitbucket, and Forgejo all inherit Git's repository-as-permission-unit unchanged: this primitive is the thing none of them dared to replace. Private files, private sub-packages, and private "branches" are the same mechanism applied at different scopes.

As data — The whole pillar is two records and two operations. The one fork that matters is what you hash once objects are encrypted; Thaddeus addresses by ciphertext (so a public mirror can store and verify a blob it cannot read) and carries a separate plaintext_id so dedup and "this is the same value, re-wrapped" survive a key rotation. 

// stored object — content-addressed by hash of CIPHERTEXT,
// so an untrusted mirror can hold & verify it without decrypting.
Object {
  id:           blake3(ciphertext)      // the address / "the name"
  plaintext_id: blake3(plaintext)       // dedup + identity across re-wraps
  alg:          "xchacha20poly1305"
  nonce:        <24 bytes>
  ciphertext:   <bytes>
}

// capability = the object's content key, sealed to ONE identity.
Capability {
  object:       plaintext_id            // what it unlocks (stable across re-wraps)
  grantee:      did:key:z6Mk…           // an identity pubkey — human OR agent (P09)
  wrapped_key:  seal(content_key, grantee.x25519_pub)   // libsodium sealed box
  granted_by:   did:key:z6Mk…
  not_before:   2026-06-22T00:00Z       // time enters here → the membrane (P02)
  sig:          ed25519(granted_by, canonical(object‖grantee‖not_before))
}

grant(obj, who)  = append a Capability wrapping content_key to who
revoke(obj, who) = new content_key → re-encrypt obj (new Object.id)
                  → re-issue caps for the remaining grantees only.
                  Old ciphertext stays addressable but is now undecryptable.

Revocation is not deletion and not a flag — it is a key rotation, which is exactly why "fire someone" and "quarantine a compromised agent" (P09) are the same one-line operation, and why a leaked old ciphertext is inert. Be precise about what that buys: rotation stops future decryption of the re-keyed object — it cannot recall plaintext a principal already decrypted and copied, and a remaining grantee who is offline at rotation still holds the old key until it re-syncs. Revocation closes the door going forward; it does not un-read what was already read. The did:key grantee here is the same portable identity that signs provenance (Pillar 04) and accrues reputation (Pillar 07): one primitive, three uses.

02

Visibility as a time-varying policy — "the membrane"

Because the policy includes time and audience, visibility can change on a schedule or an event. A single change can live in a private plane visible only to its author (an in-flight PR), be shared with a review group, be merged and released to a named set of downstream identities (the vulnerable distributions), and only then flip to public — at a chosen timestamp or trigger.

Mechanics — This delivers the coordinated-disclosure flow Git cannot: merge the fix, cut and sign the release, push it to everyone who needs patching, and reveal the source after the disclosure window closes. "Open enough" becomes directly expressible — most of the tree public, some paths private, some private only for now — all inside one repository, with no splitting and no out-of-band secret channel.

As data — A membrane is one record. Visibility (object × identity × time) is a list of grants, each a did:key plus a start time; the special public identity is how "becomes world-readable" is expressed. Crucially, a future grant is not a flag flip — because the bytes are encrypted (Pillar 01), revealing means a key-release event: at the trigger, the object's content key is re-wrapped to the public identity (a world-readable key posted to the open mirror). That single sentence is the bridge between this pillar (time) and Pillar 01 (keys), and it is why ciphertext can sit in a public mirror the entire embargo and only become readable at T. 

Policy {
  scope:    glob("src/auth/**")
  grants: [ { id: "@maintainers", from: T0 },              // private now
            { id: "@distros",     from: release(v6.7.1) },  // on release event
            { id: "public",       from: 2026-07-05T00:00Z } ] // scheduled reveal
  trigger:  timestamp | release(tag) | manual | event
}
// A grant to `public` at T ⇒ at T, re-wrap content_key to the world-readable
// key and post it to the open mirror. "Reveal" is key-release, not a flag.

Worked example — one CVE, private merge to public reveal. This is the flow Theo's "security crisis and bickering over where to store files" demands, end to end, as commands and a timeline rather than a wish: 

# 1 · open the fix in a private plane — caps wrapped only to maintainers
$ Thaddeus branch fix/CVE-2026-1234 --visibility=private --grant=@maintainers

# 2 · author commits. Each edit is an Op (P03). For an EMBARGOED op the
#     metadata is gated too — only an opaque ordering token reaches the
#     public mirror, so neither the patch nor its location/timing leaks.

# 3 · cut + sign the release; grant a capability to a NAMED downstream set
#     (Debian, RHEL, Arch maintainer keys) — to the release artifact only.
$ Thaddeus release v6.7.1 --sign --notify=@distros

# 4 · downstreams pull, build, ship patched packages — still no public plaintext.

# 5 · schedule the reveal: a key-release at the disclosure deadline.
$ Thaddeus reveal fix/CVE-2026-1234 --at=2026-07-05T00:00Z
  → writes Reveal{ scope: src/auth/**, from: "private", to: "public",
                   trigger: 2026-07-05T00:00Z, key_release: re-wrap→public }
  # at T the content key goes world-readable; the ciphertext that sat in
  # the open mirror the whole time flips to readable for everyone at once.

No out-of-band secret channel, no private fork, no "merge then scramble to publish." The window between patched and public — the window in which today's agents derive the zero-day from the diff (P3) — is a scheduled key-release, owned by the maintainer, not a race the attacker wins by reading the tracker.

The honest part — sealing the payload is not enough. A patch leaks as much through its shape as its bytes: if an op's path/symbol, author, timing, and cleartext intent (Pillar 03/04) replicate to the public mirror the instant the fix lands, an attacker reads the vulnerability's location, component, and disclosure timeline straight off the metadata — and the sealed payload is the easiest part to reconstruct once you know exactly where to look. A membrane that hides only the bytes has merely moved P3 from the diff to the metadata. So the membrane gates metadata, not just payload: an embargoed op publishes only an opaque, capability-gated ordering token to the public mirror — enough for peers to order and place it, nothing that names what or where it is — and the real fields release with the key at T. That makes the embargo genuinely leak-proof, and it names a real cost: convergence then has to run over metadata some nodes cannot read. That tension — fast convergence wants cleartext metadata; a true embargo wants it sealed — is the frontier this design owns in Part VI, not a detail it waves away.

03

An operation log with continuous convergence — branches and the repository dissolve

The change model is the operation log declared in the decision above, and it is bold about its consequences: it does not just demote branches, it largely dissolves them. As in jj, the working copy is tracked the moment you touch it and history is captured as a side effect of any action — you never stop to "stage" or curate a commit. But the deeper claim is structural. When signed operations converge continuously with CRDT semantics, there is no persistent line of development to name and babysit. There is one continuously-merging shared state, plus per-principal views over it. "Merging" stops being an event and becomes the normal condition; the rare event is a genuine conflict — two operations that contradict at the level of meaning, not whitespace (see Pillar 08) — which is surfaced for a decision instead of blocking everyone.

Mechanics — Changes are operations merged with CRDT semantics (the Delta DB approach), recorded in real time. What Git calls a "branch" becomes a lightweight, named view — a saved filter over the converging graph, scoped by capability — that you can still call main for the humans, but which is a convenience over a fundamentally different model, not a copy of the tree. And the same dissolution applies one level up: the "repository" is not a thing you clone but a capability-scoped slice of a global object/symbol graph — a named, permissioned query, not a physical pile. You do not clone a repo and hold every byte; you hold a view of exactly the parts your capabilities admit, materialized on touch (Pillar 05). The obligation to hold your history's shape — and your repo boundaries — in your head disappears, because the substrate holds them.

As data — and the answer to the riskiest interface. The hardest problem in the whole design is that CRDTs and per-object encryption seem to fight: how do you merge operations you may not be allowed to decrypt? The structural move that makes it tractable is to split every operation in two. The metadata a peer needs to order and place an op — its id, its parents (the DAG), a Lamport clock, the author, and the target path/symbol — is cleartext. The payload — the actual bytes inserted or deleted — is capability-gated ciphertext (Pillar 01). A node converges the history without reading edits it lacks the key for; to that node the path is an opaque sealed region. 

Op {
  id:       blake3(op)            // mergeable WITHOUT decryption
  path:     "src/auth.rs"          // or a symbol id under Pillar 08
  parents:  [op_a, op_b]          // the DAG — CRDT merge runs over this
  lamport:  42                    // logical clock for deterministic ordering
  author:   did:key:z6Mk…         // same identity that holds caps (P01)
  payload:  Capability-gated ciphertext   // only grantees can read the edit
  sig:      ed25519(author, canonical(id‖path‖parents‖lamport))
}
// merge is defined over { id, path, parents, lamport } — all cleartext.
// content-level 3-way conflict resolution is deferred to identities that
// hold the capability; everyone else sees a sealed, well-ordered region.

The one part that stays genuinely open is cross-visibility textual conflict resolution — when a 3-way merge needs the bytes and the merging party can't see them. Thaddeus's stance: ordering and placement always converge (cleartext metadata); content merges are escalated to a capability-holder, and to everyone else the region is simply sealed and consistent. That is a named, bounded sub-problem to prototype, not "hard research territory" left as a shrug — and it is restated as exactly that in Part VI. One deliberate consequence to flag here: cleartext metadata is what makes convergence cheap, but it is also what an embargo cannot afford to leak (Pillar 02). The default is cleartext; an embargoed op instead carries only an opaque ordering token and seals the rest — buying privacy at the price of converging over metadata some nodes can't read, the second open sub-problem Part VI owns.

04

A first-class "why" layer

Every operation carries structured provenance: the acting identity (a human, or a specific named agent), the intent or prompt that produced it, the reasoning, and links to the task or issue it serves. The "why" is stored and queryable alongside the "what."

Mechanics — An agent picking up work reads why a line exists instead of re-deriving it, and provenance is signed so the record is trustworthy rather than a vector for fabricated history. This is the durable context Entire and Delta DB are reaching for — made native to the primitive rather than bolted on beside it.

As data — Provenance is a signed record attached to an Op.id. The prompt is stored by reference, not inline — a hash plus a capability-gated pointer — so a prompt that contains secrets is not leaked into world-readable history. What signs it, and over what bytes, is explicit: the actor's key over the canonical concatenation of the fields. 

Provenance {              // attached to an Op.id (P03)
  op:         blake3(…)
  actor:      did:key:z6Mk…           // human OR agent
  actor_kind: "agent:claude-code@1.2" // or "human"
  intent:     "fix race in token refresh"
  prompt_ref: blake3(prompt_text)     // pointer; prompt stored capability-gated
  reasoning:  "refresh() re-entered before lock; added a mutex"
  task:       "Thaddeus-417"
  sig:        ed25519(actor, canonical(op‖intent‖task‖prompt_ref))
}
# "queryable alongside the diff" — concretely:
$ Thaddeus log src/auth.rs --why
  @@ refresh() … (Op 9f2c, lamport 42)
  actor   agent:claude-code@1.2  (operator: did:key:z6Mk…)   ✓ verified
  intent  fix race in token refresh        task  Thaddeus-417

Trust rule — unsigned or signature-invalid provenance renders as unverified and never counts toward an agent's reputation (Pillar 09). That is the wiring that stops the "why" layer from becoming a durable history of plausible lies (the Part VI risk): the guarantee lives in Provenance.sig and in outcomes the substrate measures itself, not in self-report.

05

A virtual, in-memory, API-first filesystem

The canonical interface is an API — read, write, list, grep an object tree — not a CLI shoved against a real disk. A virtual filesystem (the just-bash model) materializes only the paths you actually touch, lazily, in memory; nothing forces a storm of small-file writes onto APFS.

Mechanics — Working copies become cheap, unlimited, copy-on-write views: every agent gets its own instantly, with no worktree to hijack a branch and no manual sync against main. A FUSE-style mount can still present "real files" to humans who want them, but the substrate is the API, so source control runs natively inside the stripped-down sandboxes where agents already live. This is exactly what code.store demonstrates in production — in-memory writes, ephemeral branches, grep, cold storage, and repo creation in a single call.

As an API — The brief names the verbs (read, write, list, grep); here they are, modeled on and crediting code.store's production surface. "Reading and writing files should be plain API calls" (the source's exact ask) is the whole pillar in six lines: 

const repo = await store.open("acme/web")              // a capability-scoped view, not a clone (P03)
const view = repo.checkout(snapshot, { copyOnWrite: true })  // cheap per-agent COW view → no worktree (P06)
await view.write("src/x.ts", bytes)            // in-memory, no disk → no APFS small-file storm (P07, P08)
const hits = await view.grep(/TODO/, "src/**")   // grep an object tree, not a checkout
const tree = await view.list("src/")
await view.snapshot({ intent, actor })          // history as a side effect (P03) + provenance (P04)
// and the one-call create the gh CLI needs nine prompts for (P11):
const fresh = await store.createRepo("acme/new")    // or just push to a bare scope

One surface, four problems: a COW checkout is the answer to worktrees (P6), an in-memory write sidesteps the APFS trap (P7, P8), and createRepo collapses the nine-prompt gh-CLI flow to a line (P11). The FUSE mount is one optional adapter on top of this — not the substrate. These are the same method names the code.store reference card points to in Part VI: the existence proof and the proposed API are deliberately the same shape.

08

Code as a structured, queryable graph — files are a rendered view

This is the boldest break with Git, and the one no "alternative" has dared: the canonical representation of code is not text. It is a language-aware semantic graph — symbols, definitions, references, types, and the edges between them — and the file you open is a rendered view of that graph, the way HTML is a view of a DOM. The operation log (Pillar 03) records structural operations over this graph: rename-symbol, change-signature, move-definition, extract-function — not line edits over a buffer. P14 says meaning is the artifact and text is one rendering; this is the primitive that makes that literally true.

Mechanics — A rename is one operation, not a thousand-line find-and-replace, and it never conflicts with an edit elsewhere in the symbol. A signature change propagates as a typed event: every caller is a known edge, so the system raises structural conflicts (a contract actually broke) and ignores the fake ones (formatting, import order). Visibility becomes per-symbol, not just per-path — hide one function inside a public file, not the whole file. That expressiveness cuts both ways, and it is worth saying plainly: per-symbol visibility is more powerful than Git's all-or-nothing repo flag and easier to get wrong — hide a function but not its type signature or its call sites and you leak by omission; over-scope a glob and you silently hide code a teammate cannot even tell is missing. Git's model is dumb but legible; this one needs a "what can each identity see right now" view to stay safe — an owned problem, not a free win. Merges that today explode on whitespace simply do not, because the unit is the symbol, not the line. The cost is real, and larger than it first sounds: you need a language server per language to project text into graph and back, on the write path and at fleet scale — so text stays the default and the graph lights up language by language (the honest accounting is in Part VI), rather than a tax levied on every op from day one. Text diffs still exist, as a drill-down view; they are no longer the thing the system is built on. This is the second primitive every Gen-2 clone left untouched (P18): GitLab and the rest still version the file by the line, which is why they are sideways steps and not a generation — changing the unit of change is exactly the move a successor has to make and none of them made.

One edit, end to end — the five primitives are one system

The pillars above are easier to read apart than to believe together. So here is a single edit flowing through all of them at once — the connective tissue that proves this is one substrate, not five essays: 

1  view.write("auth.rs", bytes)         // P05 — in-memory, copy-on-write view; no disk, no worktree
2  view.snapshot({ intent, actor })    // P03 — captured as a side effect; no stage/commit ritual
3  → emits Op { id, parents, lamport,   // P03+P01 — cleartext metadata so peers can merge,
            payload: ciphertext }      //          ciphertext payload only grantees can read
4  → attaches Provenance { actor,       // P04 — signed why: who, intent, task, prompt_ref
                 intent, sig }
5  → Policy for "auth/**" decides       // P02 — who holds the content key now,
       who holds the key, and when      //       and who gets it at a future reveal (key-release)
6  → public mirror stores the           // P01+P02 — ciphertext is replicated immediately
       ciphertext immediately            //          but unreadable until a reveal releases the key

If the change is structural — a rename-symbol rather than a line edit — step 3's Op targets a symbol id instead of a path (Pillar 08), and the merge in step 3 raises a conflict only if a contract broke, never on whitespace. Same pipeline, finer unit. Everything downstream — the live queries of Part IV, the review-as-function of Pillar 10 — reads off exactly these records.

Part IV

04The commons

The split this brief draws between "substrate" and "commons" is a courtesy to the reader, not an architecture. The original sin of Git-and-GitHub was exactly that seam — a dumb primitive with a smart platform bolted on top, so identity, permission, review, and discovery lived outside the thing they governed and could never reach into it. Thaddeus has no such seam: objects, capabilities, identity, history, structure, and query are one substrate, and "the commons" is simply that substrate deployed as a hosted, federated home. Five more primitives carry it up to the social layer — built for agent scale and for an ecosystem that will live in many places at once.

06

A platform built for agent throughput and real discoverability

The remote is API-first and engineered for the load agents actually generate — the code.store numbers are the existence proof — not a human-era monolith. Creating a scope is one call or a bare push. CI/CD runs against ephemeral in-memory checkouts, so a clean environment is cheap instead of a 140-second APFS tax.

The infrastructure target, concretely — the bar is code.store's production envelope: ~9M repos in 30 days, ~15K repos/min for three hours, zero downtime, on an in-memory, API-first engine that scales horizontally where GitHub's Ruby monolith buckles at ~20M new repos (P9). Thaddeus takes that infrastructure wholesale and changes one thing — Git is no longer the primitive underneath it. code.store is still a git cloud; Thaddeus's source of truth is the operation log — the clean break declared at the top of Part III — and Git is only an export bridge. The same numbers, with Git's filesystem and snapshot model out of the hot path.

Repos are created in code — not provisioned through a wizard. A scope is an object you allocate from the API, exactly as code.store does it, so an agent fleet stands up thousands of them in parallel, in memory, inside that throughput envelope — the nine-prompt gh-CLI flow (P11) gone entirely: 

// a scope is allocated in code: one call, in-memory, ~ms — no UI, no provisioning
const store = new Thaddeus()
const scope = await store.createRepo("acme/agent-run-8f2a")

// or skip creation entirely — a bare push brings the scope into being (Forgejo's trick)
$ Thaddeus push acme/agent-run-8f2a

// fleet scale: thousands of scopes in parallel — the code.store envelope, no downtime
await Promise.all(runs.map(r => store.createRepo(r.id)))

Mechanics — Because the underlying model is structured — typed snapshots over a semantic graph, releases, signed provenance — discoverability becomes query, not scroll. "The change from last March that touched this symbol," "every change between these two releases," "the change immediately after this tag" are answerable directly. Release pages, search, and history stop being infinite-scroll exercises layered on a model that can't answer the question. This is the entry point to a far larger capability — the codebase as a live database — specified in Pillar 11.

Answering the diagnosis's own broken queries. P10 named three things GitHub and GitLab cannot do; structured history answers each as a one-liner instead of a clone-and-grep: 

# P10: "find a commit from last March" — infinite scroll, no date filter today
$ Thaddeus log --path src/auth.rs --since 2026-03-01 --until 2026-03-31

# P10: "what changed between two releases" — no such view today
$ Thaddeus diff --from v6.6.0 --to v6.7.0

# P10: "the change immediately after this tag" — GitHub has parent, no CHILD link
$ Thaddeus next <tag>        # step forward to "what broke right after this release"

And because a release is a typed object, the release page stops being infinite scroll with a meaningless "88% complete" and the date buried under the changelog (P10's exact complaint). The useful data is first because it is structured:

Portable identity and federated reputation as a protocol

The one thing you cannot rebuild as a walled garden is the home. So identity is a portable, self-owned cryptographic identity — the same identity that signs your changes and holds your decryption capabilities — and contribution history is a signed, aggregatable graph that travels across instances.

Mechanics — Self-hosted and hosted instances federate a shared identity and reputation layer, so "click a username, see everything they've built" survives even though the code itself lives in many places. This will not undo the fracturing already underway — graduation is graduation — but it is the only design that lets a scattered ecosystem still share one social graph instead of dissolving into mutually-illegible islands. And critically, the identities in this graph are not only human (Pillar 09).

As data — An identity is a did:key — the same self-owned key that wraps capabilities in Pillar 01 and signs provenance in Pillar 04; three uses, one primitive. Reputation is not a number a server hands out; it is the set of signed contribution records a verifier can gather from any instance and check for itself. 

Identity = did:key:z6Mk…   // self-owned; signs caps (P01) + provenance (P04) too

// published at any instance, verifiable anywhere — no central graph owner:
Contribution {
  subject:   did:key:…                 // who
  repo:      "forgejo.example/acme/web"   // where it lived
  ref:       snapshot_id
  kind:      "merge" | "review" | "release"
  at:        2026-06-22T…
  host_sig:  ed25519(instance_key, …)  // the instance attests it happened there
  subj_sig:  ed25519(subject, …)       // the author co-signs it
}

Aggregation — a profile is just the set of Contribution records signed by that identity, gathered from wherever they live. A verifier checks subj_sig (authenticity — the subject really claims it) and optionally host_sig (it really happened on that instance). No central server owns the graph; aggregators are untrusted indexers over signed records, so instance B can honor a contribution made on instance A without trusting B. That is what turns "click a username, see everything" from a wish into a verifiable protocol — and it is why the honest network-effect risk in Part VI can stand without the mechanism also being vague.

Governance, not just hosting — Portability is also the answer to lock-in (P19) and the rug-pull (P20). Because identity, history, reputation, and the code itself are content-addressed and federated, leaving any one host is an export, not an amputation — there is no proprietary issue tracker holding your project hostage the way Bitbucket binds you to Jira, and no owner who can take the project private and force a community fork the way Gitea did to the people who built it. Thaddeus's open, self-hostable core under transparent, nonprofit-style stewardship — the Forgejo/Codeberg lesson, including their Mastodon-grade incident transparency — is a design requirement, not a feature: when the substrate is portable and the governance is accountable, "what happens when the steward's incentives change" has a real answer — you move, and your home moves with you.

09

Agents as first-class principals — identity, reputation, capability, economy

If a million agents write the code, they are not clients of the system; they are its citizens. Thaddeus makes every agent a first-class principal with its own cryptographic identity, distinct from the human who operates it. A change is signed by the agent and attributed to its operator — so accountability is precise instead of laundered through a human who never read it. This is the missing half of P16 and the reason the whole design is possible.

Mechanics — Four things attach to an agent identity. Reputation: a score derived from verified outcomes — merge rate, regressions, reverts, audit-pass rate — that feeds directly into merge policy (Pillar 10), so a high-reputation agent earns autonomy and a misbehaving one loses it. Capability: scoped permissions over the graph (which symbols and paths, which actions) plus rate and spend budgets — an agent may touch these symbols, open N changes an hour, cost $X — enforced by the substrate, not by hope. Revocation: a compromised agent is killed by rotating its keys, and its operations are quarantined from the converging state — the concrete answer to the npx-from-a-skill.md and is-odd-with-a-zero attacks, and the blast-radius control the git push -o RCE and the squatted-tanstack exfiltrator (P21) had nowhere to live in on GitHub. Economy: a change can carry a paid, third-party verification attestation — pay a verified auditor (an agent or a service like Socket, which already catches npm exploits before npm does) to read it and sign a safety verdict, generalizing the "50 cents to audit a release" idea into a substrate primitive and turning supply-chain trust from a hope into a priced, signed fact that travels with the dependency. Authorship becomes signed, scoped, priced, and revocable.

Review as policy, proof, and reputation — not one human reading a diff

The pull request was a human-in-the-loop ritual: one reviewer reads the changed lines and clicks merge. At a million authors it does not scale, and rendering the diff more nicely does not change that (P15). Thaddeus replaces the ritual — keeping the contract — by making merge a function, not a gesture. A change merges when it satisfies (a) automated verification — types, tests, property checks, security policy-as-code, and where it matters, formal proof — and (b) the author's signed identity and reputation tier (Pillar 09). Humans set the policy and adjudicate the exceptions. They do not gate every change by reading it.

Mechanics — The reviewable artifact is no longer a wall of text. It is a semantic and behavioral diff built on Pillar 08: "this change alters these 3 symbols, changes this type's contract, adds these 2 behaviors verified by these tests, and touches no auth-scoped code." A human reviews claims and proofs about a change and drills into raw lines only when a claim looks wrong. Because operations converge continuously (Pillar 03), integration is literal and constant; a "release" is a policy event — a visibility and verification threshold crossed (Pillar 02) — not a human mashing a merge button. Trust tiers make it concrete: a high-reputation agent's change that passes every gate merges under policy without a human reading it; an untrusted agent's change to sensitive symbols always escalates to a person. One principle has to survive the automation, though: a human keeps the standing right to read any change and veto it, even one a green policy would merge — the policy function sets the floor for what merges automatically, it never strips a person's authority to say no. Retiring the diff as the mandatory gate must not retire the veto. This is what review looks like when machines write the code and humans own the rules.

The codebase as a live, subscribable database

Discoverability-as-query (Pillar 06) is the small version of a much larger claim: the codebase is a live database, and you can query it, subscribe to it, and run policy over it continuously. Not "search history better" — query the present semantic state of the code the way you query Postgres, get answers in milliseconds, and stand up triggers that fire on meaning. This is the payoff of storing code as a graph (Pillar 08) over a signed operation log (Pillar 03) with capabilities (Pillar 01) and provenance (Pillar 04): every dimension the query needs is already first-class.

Mechanics — A query layer spans the semantic graph, full history, provenance, and capabilities at once — enabling questions impossible in Git or GitHub today: "every function that still calls this deprecated API, across all history." "All code an untrusted agent touched in the last hour." "Every place this secret-capability is required, and who holds it." "Show the behavioral delta this agent's changes introduced this week." Live subscriptions fire on semantic events — "tell me when anyone changes the signature of this public function" — instead of coarse push webhooks. And policy runs as standing queries: "no untrusted agent may modify auth code" is not a CI script that runs late but an invariant the substrate enforces as changes converge. The codebase stops being an inert pile you scroll (P17) and becomes an organism you can interrogate and govern in real time.

The hard problems we are choosing to take on

the frontier, owned — not caveats

A field-defining system is defined as much by the problems it volunteers for as by the ones it solves cleanly. These are not reasons to hesitate; they are the work. Naming them precisely is how you start.

CRDTs and per-object encryption have to be made to coexist

Merging operations you may not be allowed to decrypt, and defining conflict semantics that cross visibility boundaries, is the riskiest interface in the design. The shape of the answer is the Op split shown concretely in Pillar 03: operation metadata (id, parents, Lamport clock, author, target path/symbol) stays cleartext so the DAG converges for everyone, while the payload stays capability-gated ciphertext — you order and place an operation without reading it. What is genuinely open is narrower than "hard research territory": it is one named sub-problem — cross-visibility textual 3-way conflict resolution, the case where resolving a content conflict needs bytes the merging party can't see. Thaddeus's stance is that ordering and placement always converge, and content merges escalate to a capability-holder while the region stays sealed and consistent for everyone else. That bounded sub-problem is the first thing to prototype, and the bet the whole substrate rides on. A second, separable question is scale rather than correctness: Delta DB proves operation-CRDT sync for a handful of collaborators in one session, not a planet-sized op-DAG converged by a million principals. Whether convergence stays sub-linear per write as the DAG grows — and how much metadata each peer must hold to converge — is unproven at that scale, and belongs to this same frontier, not to the list of solved imports.

The embargo has to seal metadata, not just bytes

The membrane (Pillar 02) is only real if an embargoed change hides its existence and location, not merely its payload. But the very op split that makes convergence cheap (Pillar 03) keeps path/symbol, author, timing, and intent in cleartext — which, for a security fix, is most of the disclosure. The owned problem: a metadata-private op that publishes only an opaque, capability-gated ordering token to the public mirror — enough to order and place the change, nothing that names it — with the real fields released at reveal. The frontier is making CRDT convergence run over metadata a node cannot read: ordering must stay deterministic and verifiable from the token alone. This is the bounded form of "encrypt the metadata too," and it is what turns P3 from relocated into resolved.

An append-only log that is the source of truth has to be able to forget

The operation log never deletes by design — revocation is re-encryption, not removal (Pillar 01), and old ciphertext "stays addressable." Across a million agents writing continuously, with a signed provenance record on every op (Pillar 04), that is monotonic growth forever, and both convergence and snapshot-materialization cost rise with log size. So compaction is not optional housekeeping; it is a primitive the design still owes: epoch checkpoints that let a peer treat history before a watermark as a sealed snapshot, op-log GC for superseded and dead-ciphertext objects, and a stated retention model. A source of truth that can only grow is, on a long enough timeline, a source of truth that stops converging.

The semantic graph needs a language server per language

Treating code as a structured graph (Pillar 08) means projecting text into symbols and back for every language you support — a real cost, and a larger one than "editors already do this" admits. An editor runs a language server for one developer's one project; Thaddeus would put projection on the canonical write path for a fleet of agents, and real servers (rust-analyzer, tsserver, gopls) cost seconds to cold-index and hundreds of MB to GBs of RAM per workspace. The honest strategy follows from that: text is the default and the universal fallback (you are never worse than Git), and the structured graph is an opt-in superpower lit up language by language and kept warm where it earns its cost — not a tax on every op in every language from day one. Still a finite, fundable engineering march — but a march, not a free import.

The threat model is agent-scale, and security is now a feature of the substrate

Once agents hold decryption capabilities, the question is what they can exfiltrate — and here precision matters, because it is easy to over-claim. Sandbox attestation (TPM/SGX/TDX/Nitro-style) can prove which code image is running on attested hardware, so capabilities can be withheld from unattested agents; what it cannot do is prove approved code won't leak — once an agent legitimately decrypts, the plaintext is in its address space and can leave by any permitted channel. So the honest guarantee is "bounds which agent may decrypt," not "prevents exfiltration once it has." The controls that actually bite are the ones the substrate can enforce around that fact: least-capability scoping, tight budgets, fast revocation (Pillar 09), and supply-chain provenance that travels with dependencies. Source control that is also a security boundary is the right framing — owning exactly how far the boundary reaches is part of owning it.

Key management at human (and agent) scale is the usual landmine

Rotation, recovery, device loss, onboarding, and now per-agent key issuance and revocation all become security-critical. The mechanism is deliberately off-the-shelf, not invented here: each identity has per-device subkeys, so a lost laptop revokes one subkey rather than the identity; account recovery uses threshold/social recovery (M-of-N guardian keys, Shamir-split) so device loss ≠ data loss; and organizations hold an escrow capability, so an employee leaving triggers a re-wrap to the org key (the concrete form of "firing = key rotation" from Pillar 01) instead of a scramble. Two of these carry honest tradeoffs the design must own rather than paper over. The org escrow is, by construction, a key that can decrypt the org's objects — a master key — so it quietly reintroduces the single point of catastrophic compromise that ciphertext-at-rest (P21) was meant to remove; it is only safe if it is itself threshold-split and its every use is logged immutably to the op log, never a quiet standing super-grantee. And the failure mode is harsher than Git's: lose the recovery quorum and the bytes are mathematically unreadable, not "recoverable from a reflog." So the real risk is not the primitives — device subkeys, Shamir recovery, org escrow all ship today — it is making the whole lifecycle invisible and survivable; treat it as a product problem, with agent key lifecycle a first-class case, not an afterthought.

The on-ramp: an existing repo gets in, and git push works day one

The decision to leave Git (Part III) is a clean break in the primitive, not a flag day for users. A Git-compatible gateway speaks the smart-HTTP/SSH protocol, so git clone and git push work unchanged against a Thaddeus scope. Import is a fixed mapping: each commit → a snapshot, each blob → a (cleartext-capability) object, each branch → a named pointer/view, history → a linear op-log. New Thaddeus-native features — capabilities, the membrane, the why-layer, structural ops — are additive and simply invisible to a plain Git client, which keeps seeing commits and blobs. This is the same one-way interop Delta DB already demonstrates with Git and Forgejo demonstrates by accepting GitHub-format Actions YAML 1:1. The clean break stays available for anyone who wants the operation-first core with no Git baggage; the gateway just means migration is an import, not an amputation. The deeper on-ramp is human, not technical: because the Git-compatible view shows only commits and branches, a developer feels none of the native value until they cross into capabilities, the membrane, and code-as-graph — and then several unfamiliar nouns arrive at once, exactly at the point of value. The honest task is a graduated path where a team keeps its familiar nouns and gains one native capability at a time, so the climb is incremental rather than a cliff; designing that ramp is as load-bearing as the gateway itself.

The community problem may outrun any protocol

Federated identity (Pillars 07, 09) is the best available answer, but the shared home GitHub provided is partly a network effect, not a feature. The fracturing has already begun; a protocol can preserve the social graph without recreating the gravity. We choose to build the graph anyway, clear-eyed about what gravity it cannot restore — because the alternative is mutually-illegible islands.

Verifiable provenance, or a durable history of plausible lies

The "why" layer (Pillar 04) and agent reputation (Pillar 09) are only as good as they are verifiable. Provenance must be signed, and outcomes must be measured by the substrate, not self-reported — or a durable record of intent becomes a durable record of fabrication. Two attacks follow from the fact that did:key identities are free and unlimited: sybil / reputation-farming (mint throwaway identities, farm green outcomes on disposable scopes, then spend the reputation on one malicious change) and a buyable audit oracle (stand up or bribe a reputable-looking auditor to sign "safe" on your own payload). So the design owes more than signatures: a scarce or staked root for identity, anti-farming on the outcomes that feed reputation, slashing for auditors who attest falsely, and a standing rule that reputation alone never unlocks auto-merge on sensitive symbols (Pillar 10) — a human stays in that loop by policy. Audit is an economy primitive in Pillar 09 rather than a trust assumption for exactly this reason — but an economy still needs its anti-fraud built in.

Part VII

07Beyond the complaints — where this goes in ten years

P1–P17 are the entry wedge: every one is a pain that exists today, and a system that fixes them all is already worth building. But a list of 2026's complaints can only describe a better version of 2026. The reason to pour a new foundation is what it makes possible — capabilities no Git workflow could host, because they require exactly this substrate. The matrix is the floor. This is the ceiling, and it is high.

Continuous formal verification as the merge gate

When merge is a function of proof (Pillar 10) and code is a typed graph (Pillar 08), formal verification stops being an academic luxury and becomes a standing gate: critical invariants are proven to hold on every converging operation, not hoped to hold after a test run. "This change cannot violate this safety property" becomes a checkable, blocking claim — for a million agent authors, the only kind of trust that scales.

Code as generation-on-demand, not stored text

If the truth is an operation log plus a spec and a semantic graph, "the code" need not be stored as text at all. It can be generated on demand — rendered to whatever language, style, or target a reader or agent needs, from a canonical structured source. The file becomes the cache, not the original. Thaddeus is the first model where that is even expressible, because it already treats text as a view.

A shared substrate across organizations

When identity, reputation, and capabilities are cryptographic and portable (Pillars 07, 09), they do not stop at a company boundary. An agent's reputation, a dependency's audit attestation, a capability to read one private symbol — all can be honored across organizations on a federated substrate. The unit of collaboration grows from the repo to the ecosystem, with permission and provenance intact the whole way.

Audit and compliance for autonomous code

A world where agents author most code will demand regimes that do not exist yet: who changed this, under what authority, verified by whom, and can you prove it years later. Thaddeus's signed operations, agent identities, scoped capabilities, and standing-query policy (Pillars 04, 09, 11) are exactly the primitives such regimes will be built from. The system that versions meaning is also the system that can be held accountable for it.

None of these are bolt-ons. Each falls out of the same decisions — operations over snapshots, meaning over text, principals over users, policy over rituals — that the diagnosis forced. That is the signature of a right foundation: the future you did not design for is already reachable from it.

Part VIII

08Who pays for the commons

Every part before this one answers what to build and whether it can be built. None answers the question that decides whether it still exists in ten years: who pays for it, and how does it stay un-ownable while they do? For Thaddeus that is not a business-school annex to the architecture — it is forced by it. A system whose entire pitch is non-capturable, federated, can't be rug-pulled (Pillar 07, P20) cannot be funded by a model that quietly hands one company the power to rug-pull it. The business model is a load-bearing wall, and it has to be poured from the same concrete as the rest.

Start with the instinct everyone reaches for: do what Forgejo does. But Forgejo names two things people blur together. Forgejo is the software — GPL, forkable, free. Codeberg is a German nonprofit (Codeberg e.V.) that runs the hosted instance on donations and memberships. Codeberg is not a business; it is a governance structure that has chosen, deliberately, never to field a market-rate team. To admire it is to admire its non-capturability — not its revenue, because it has almost none. The two systems that actually fund large teams sit at the opposite pole: GitLab, an open-core public company, and GitHub, proprietary and sold to Microsoft. So the real problem statement is sharper than "be open source." It is: win Codeberg's trust and GitLab's revenue, while escaping Codeberg's poverty and GitHub's capture. The way through is to stop hunting for one license that does everything and split the system into three layers, each with its own answer.

Think in three layers, not one license

open where it builds trust · paid where openness doesn't erode value

The protocol must be the most open thing you own. Portable identity and federated reputation (Pillar 07) do not function if the protocol is capturable — a standard nobody can trust to stay neutral is not a standard. Charging for it would poison the very adoption that makes the commons worth anything. You make your money one layer up, and the reason that works is the thing every "open source but only we charge" plan gets backwards.

The license was never the moat

The fear underneath "open but only we monetize" is that openness gives the business away. It doesn't — and the proof is the incumbent. Git is MIT/GPL and infinitely forkable, and GitHub still built a multi-billion-dollar business directly on top of it, with no license trick at all. The value was never the primitive; it was the commons — identity, the network, reliability, the home. That is defended by gravity, not by a license, and gravity does not weaken because the substrate beneath it is open. Thaddeus's gravity is the federated identity-and-reputation graph (Pillar 07) — the same network effect that made a GitHub profile worth having (P13). The one place a protective license genuinely earns its keep is narrow: stopping a hyperscaler from cloning your hosted platform code and out-distributing you with it — the AWS-versus-Elasticsearch move. So put the platform server under a source-available license (FSL or BSL) and keep the protocol and substrate genuinely open. Protection exactly where it is threatened; openness everywhere it compounds.

Pricing that grows with agents, not seats

why the unit of billing changes when the authors are machines

GitHub and GitLab charge per human seat. That model does not survive the world this brief describes. When the primary authors are a million agents (Pillar 09), a per-seat price is both unchargeable and backwards — it taxes the humans while the agents, which do the work and burn the resources, ride free. The good news is that Thaddeus's design already carries the model that replaces it; metered budgets and paid attestations are primitives, not afterthoughts.

  • Consumption. Storage, throughput, and in-memory CI compute (Pillars 05, 06). You bill for work done, not for chairs occupied — and agents do far more work than there are chairs.
  • Marketplace take-rate. The paid third-party audit-and-attestation economy (Pillar 09) — Socket's "50 cents to audit a release," productized — taking a cut of every verification that travels with a dependency. A Stripe-shaped line that scales with ecosystem activity, not logins — but be honest about size: this is strategic upside, not a co-equal pillar, and for years it is immaterial next to hosting and enterprise. Hosting + enterprise are the core; the marketplace is the option on a much larger future.
  • Metered agent budgets. An agent already runs under a spend cap it cannot exceed (Pillar 09). Billing is therefore native — the budget that bounds a compromised agent's blast radius is the same meter that prices a well-behaved one.
  • Enterprise. Compliance and audit for autonomous code (Part VII), SSO, policy-as-code, support and SLAs — the layer companies have always paid for and always will.

The model and the thesis point the same way

This is the part to be excited about. This revenue grows with agent activity — the exact wave the whole company is betting on. Seat-based pricing fights the agent thesis, because every new agent is a cost the price cannot capture. Consumption-plus-take-rate pricing is the agent thesis: the more code a million agents write, audit, and ship, the more the substrate earns. The business compounds on the same curve as the bet — an alignment a per-seat incumbent cannot reach from where it stands.

One honesty the excitement must not skip: this only compounds upward if each marginal op is gross-margin positive. An in-memory, per-object-encrypted, graph-indexed, store-forever substrate costs materially more to serve than a disk-and-Git incumbent, and agents are relentless cost-minimizers — so "revenue grows with agents" is a claim about margin per op, not just volume. If cost-per-op ever exceeds price-per-op, the same curve runs in reverse and scale deepens the loss. The model is right; it is not free — and engineering the substrate's cost structure (compute, perpetual storage) down to where the curve points up is part of the bet, not a footnote to it.

The model has to honor the thesis

the capture tension, and the commitment device that dissolves it

One contradiction will quietly kill this if it is not designed against from the start: venture funding creates capture pressure, and the product's whole promise is non-capturability. The graveyard is recent and well-populated. Elastic, HashiCorp, Redis, and MongoDB all began open, took money, then relicensed under growth or hyperscaler pressure — and each move spawned a hostile fork that walked away with the community's trust: OpenSearch, OpenTofu, Valkey. For an ordinary infrastructure company a relicense is a bruise it survives. For Thaddeus it is fatal, because "you cannot be rug-pulled" is not a feature here — it is the reason a developer moves their identity off GitHub and onto your commons in the first place (P13, P20). Betray it once and the most sophisticated users — the ones who onboarded because they believed the promise — are precisely the ones who leave, and they leave with the identity graph that was the whole moat.

So the credible-commitment device cannot be legal boilerplate bolted on at Series B; it is part of the architecture, poured at incorporation. A foundation that irrevocably holds the protocol and the trademark, so no future board can take them private the way a for-profit owner took Gitea private (P20). A source-available license that time-bombs to true open source on a fixed clock (the BSL pattern), so openness is guaranteed by a calendar rather than a pinky-swear. A charter that outlives the cap table. Structurally this mirrors the systems that got it right — the protocol-plus-commercial-steward split of Matrix and Element, of Mastodon, of the Ghost Foundation — where a neutral nonprofit owns the standard and a company runs the hosted commons on top of it and employs the people who build both. To the right investor this is not a concession; it is the moat stated plainly: trust and federation are the product, and the foundation is what makes them defensible. Be clear-eyed that it is also a real trade, not a free lunch: it permanently forfeits the lever the capture playbook relies on — you can never relicense to defend margin, and you pre-commit the open clock by calendar. For this product that price is worth paying, because the promise is the product — but it should be chosen with eyes open, not sold as upside with no cost.

The business has the same shape as the substrate

This is why the model is not an annex to the design — it is the design, one level up. The substrate works because a dumb, open primitive cannot be captured while a smart platform pays the bills, with capabilities keeping the seam between them honest. The business works the same way: an open protocol that cannot be owned, a company that runs the hosted commons and the marketplace on top, and a foundation that guarantees the first can never be eaten by the second. Pour the foundation, then build the commons on it — in the legal structure exactly as in the code.

Closing

09Pour a new foundation

Git isn't perfect — it's barely even good. We've dealt with it for decades, so it's hard to see the cracks. Be more introspective. We can do so much better.

We have lived inside Git's assumptions long enough to mistake them for laws of nature. They aren't. "Open source" need not mean 100% public 100% of the time. Permission belongs on the change, not the repository. History should record why, not just what. Code is meaning, not text. Working copies should be free and infinite. The authors are agents, and agents are principals — with identity, reputation, and a bill. Review is policy, not a human squinting at 11,000 lines. None of these are tweaks to Git. Each one breaks something Git holds sacred, and together they describe a different thing entirely.

Picture the decade this unlocks. Code that defends itself: secrets that were never plaintext, security fixes that ship to the vulnerable and reveal themselves only after the world is patched. A million agents building in parallel, each under a signed identity, scoped capabilities, a reputation it can lose, and a budget it cannot exceed. A codebase that is a living, queryable organism — you ask it what it is, subscribe to how it changes, and stand policy over it that simply holds. One portable identity you carry across a federated commons, so the home survives even as the code scatters. That world is not a fantasy. Every piece of it already runs somewhere; what is missing is the foundation that composes them. The diagnosis is specific, the design answers it line by line, and the only thing left undone is the building.

This was economically absurd a year ago and is merely hard today — and "hard" is what agents are for. So pour the foundation. Take the clean break the last generation never got to take. Build the substrate, then the commons on top of it, and let Git become the export format of an era we finished.

Git versioned files for one human. Thaddeus versions meaning for a civilization of agents. That is not a better Git. It is what comes after it.

Appendix

AThe primitives, named

A quick glossary of the building blocks referenced throughout — both the ones this design introduces and the existing systems it stands on.

Object
An encrypted, content-addressed value. The address is blake3(ciphertext) — so an untrusted mirror can hold and verify it without decrypting — while a separate plaintext_id preserves dedup and identity across re-wraps. See the Object record in Pillar 01.
Capability
An object's content-encryption key, wrapped (libsodium sealed box) for one identity's did:key. Holding it = the right to decrypt that object. grant appends a Capability; revoke rotates the key and re-wraps only for the remaining grantees, leaving the old ciphertext inert. Not a repo-wide ACL. See the Capability record in Pillar 01.
The membrane
The visibility policy expressed as (object × identity × time) — concretely a Policy record: a list of {identity, from} grants plus a trigger. A grant to the special public identity at time T is a key-release event (re-wrap the content key to a world-readable key), not a flag flip — which is how ciphertext can sit in a public mirror until T and only then become readable. See Pillar 02.
Operation log
The source of truth: a signed, CRDT-ordered stream of Op records. Each Op carries cleartext merge metadata (id, parents, Lamport clock, author, path/symbol) and a capability-gated ciphertext payload, so peers converge history without reading edits they lack keys for. In Thaddeus the operations are the primitive and everything else is derived — the inversion of Git. See Pillar 03.
Snapshot
A content-addressed, immutable state of a value — in Thaddeus a derived, cached projection of the operation log for fast reads, verification, and migration, not the primitive. Replaces the manual commit; "branches" become named, capability-scoped views over the converging graph.
Semantic graph
The canonical form of code: symbols, definitions, references, and types with the edges between them. Operations are structural (rename-symbol, change-signature), and the text file is a rendered view of the graph — meaning is the artifact, text is one projection.
Principal · identity
Any first-class actor with a self-owned cryptographic identity — a did:key, human or agent. The same key wraps capabilities (Pillar 01), signs provenance (Pillar 04), and accrues reputation (Pillar 07): one primitive, three uses. Reputation is a set of signed Contribution records gathered from any instance and verified by checking subj_sig (and optionally host_sig) — no central graph owner.
The "why" layer
A signed Provenance record attached to an Op.id — actor, intent, reasoning, task link, and a prompt_ref (a hash/pointer, not inline text, so prompts with secrets don't leak). Queryable alongside the change (Thaddeus log --why); unsigned or invalid provenance renders unverified and never counts toward reputation. See Pillar 04.
Merge-as-function
Integration as a decision computed from automated verification (types, tests, property checks, proofs, policy) plus the author's reputation tier — not a human reading a diff. A "release" is a policy/visibility event, not a button.
Virtual filesystem
An in-memory, API-first file interface that materializes only touched paths lazily, gives every principal a cheap copy-on-write view, and never requires a real OS or disk.
The live codebase
The code as a queryable, subscribable database over the semantic graph, history, provenance, and capabilities — answering cross-cutting questions, firing on semantic events, and enforcing policy as standing queries.
