What is genuinely well-built

  • Authorship cannot be forged — ops are signed over domain-separated canonical bytes and OpLog.ingest rejects any unverifiable op (oplog.ts:191-193), so a delegate can't push an op claiming the owner as author. The key escalation path is closed.
  • Uniform signature discipline — distinct domain tags per record type, canonical JSON that rejects NaN/Infinity/non-UTC, fail-closed non-throwing verifiers everywhere.
  • Path traversal is closed — FileBackend percent-encodes each key into one flat filename; staging + atomic rename gives per-key crash atomicity; glob matchers fail closed on .. in three independent places.
  • AEAD hygiene — XChaCha20-Poly1305, fresh random 192-bit nonce + fresh content key per put/rotate; no nonce-reuse risk. did:key derives the verifying key from the DID itself, so a DID can't be spoofed.
  • Single-node concurrency is reasoned — per-repo promise-lock, single-flight registry cache, TOCTOU closed inside the lock, frozen deep-copy of a registered delegation so a caller can't widen a verified grant.

3. Is it usable? (Product/PM lens)

Headline: Thaddeus is an extraordinary substrate wrapped around a product that does not yet have a job. A disciplined developer can reach a first published change today — but three things block adoption.
  1. No acute wedge. Every headline capability is a platform feature that pays off only inside a team that already adopted the whole stack. Nothing solves a screaming today-problem for a single developer better than git init, and the switching cost is total (no git interop, no hosting, no ecosystem).
  2. You cannot version the directory you're standing in. The core git init gesture doesn't exist — thaddeus init only mints an identity (run.ts:625). To version existing code you must create an empty remote, clone it into a new dir, and move files in. This inverts muscle memory on the very first command.
  3. The documented first-run path has a copy-paste break. getting-started.md §4 says clone acme/web → ./acme/web, but the code creates ./web, so the cd fails. (As analyzed, the marquee query/watch/reveal verbs were also not wired into the CLI — but that shipped in 0.1.6, so the differentiator is now reachable. The remaining PM gap is positioning/discoverability, not existence.)

The reframe that unlocks a first user

The strongest job hiding in here is agent-governed code: "let an autonomous agent push — cryptographically bound to a scoped, budgeted, revocable delegation, with a signed why on every change and a standing human veto." That is a 2026 problem people actually have (fleets of coding agents with no real permissioning), and everything for it is already built and passing as a test (one-edit-end-to-end.test.ts:177). It's buried as bullet 8 of getting-started. Position Thaddeus as "the permission layer for AI agents committing code," with post-git as the substrate that makes it possible.

The Backend question (load-bearing)

The interface is put / get / list / delete. An S3 backend can mechanically implement it — but that's the wrong question. The interface conflates two key classes:

  • Immutable content-addressed blobs (obj/<blake3>, op/<blake3>) — write-once, concurrent writers write identical bytes, last-writer-wins is safe. S3 is perfect here.
  • Mutable registers (view/main, current/, cap/, meter/, rep/, meta/) — read-modify-write, whose only concurrency control today is the single-node event loop + an in-process lock.

The interface exposes no primitive that could replace that lock across processes — no put-if-absent, no compare-and-swap on a version/etag, and get returns bare bytes with no version to compare. So a second node gives you lost lands on main, double-spent budgets, and revocation races, undetectably. The fix isn't the storage engine (S3 gained conditional writes in 2024; SQLite has transactions; FileBackend can emulate with O_EXCL) — it's the contract, which must grow roughly: 

putIfAbsent(key, bytes): Promise<boolean>        // create-once
get(key): Promise<{ bytes, version } | undefined> // versioned read
putIfMatch(key, bytes, expectedVersion): boolean  // compare-and-swap

This is issue #68 (define CAS semantics) and it is the hard prerequisite for any multi-node work in any language.

Rust or Go — the recommendation, with triggers

Evidence the language is not the bottleneck: the compiled Bun CLI is ~47 ms warm (a git-like tool is already fast; Rust/Go buys binary size, not latency); the crypto core is real work but conflicts()/materialize()/graph re-extraction are seconds-to-minutes and are pure TS algorithm bugs; and the multi-node problem is a coordination problem (CAS + externalized state) that Go's goroutines do not solve. lazythad in Rust is the right call and good code (1,320 LOC, 7 unwraps, tolerant wire decoding, clean split) — a TUI over an HTTP boundary is the ideal Rust target; keep it.

8. ThadHub — detailed plan

What it is: a hosted, multi-tenant home for Thaddeus repos — it runs @thaddeus.run/server at scale, gives every repo/identity/release a stable URL, and provides the web surfaces GitHub taught us to expect (browse, discovery, provenance, review/veto queue, reputation profiles, agent management, releases) while holding no private keys, storing only ciphertext + self-verifying signed metadata, and letting any user walk away with code and reputation fully intact.

The inversion that defines the product: GitHub owns the identity graph, the social graph, and the canonical copy. ThadHub can own none of them without betraying the manifesto. ThadHub is not a database of record — it is the fastest, best-lit mirror and index of a substrate whose truth lives in the artifacts themselves. "GitHub is where your code lives; ThadHub is where your code is currently most conveniently mirrored." That word "currently" is the entire product ethic.

The load-bearing seam: what a ciphertext host can and can't show

The server holds ciphertext for file contents, but a large amount of structural metadata is cleartext — and that seam is what makes a useful hub possible.

Cleartext (no keys needed) Never visible to the hub
The op DAG (ids, author DIDs, lamport, parents), and crucially file paths (op.path is read by the delegation gate); branch names + heads; provenance "why" text; veto reasons; release tags/notes/manifests; grant scopes; reputation profiles; owner DIDs. The bytes of any file object — objects are ciphertext, the content key is sealed per-reader to their X25519 key. Only a capability-holder unseals.

This yields three honest visibility tiers, defined by who holds a capability for the content key, not a server ACL flag:

  • Public repo = "a capability was issued to a world-readable identity." ThadHub is then a designated reader and can render fully (code, blame, why, search) server-side — a GitHub-grade experience with the ciphertext-at-rest property unchanged.
  • Private repo, logged-out visitor = the cleartext skeleton only: repo exists, owner DID, branch names, commit-graph shape, file paths, timestamps, releases, contributor reputation — no contents, no diffs. A genuinely novel, honest UX.
  • Private repo, authorized reader = decryption happens client-side in the browser with a key the hub never receives (the hard key-custody problem — see risks).

Architecture — a shell around the existing server

ThadHub is not a rewrite; it's a deployment + frontend around createServer(config), which is already stateless over a shared Backend. The web frontend is a read-mostly client of the same HTTP API the CLI uses; every write (veto, release, grant) is signed client-side, so a stolen session cookie can't forge a signed op.

  • Namespacing — the server already scopes repos under repo/<name>/ and already tolerates / in names, so @alice/web → alice/web with zero core changes. New: a handle→DID registry as a signed HandleClaim record (first-claim-wins within an instance, but portable — the DID is the real identity).
  • Auth — did:key challenge sign-in → short-lived session cookie for read/personalization only; writes stay individually signed. Ship the replay-nonce cache (#61) before public launch.
  • Discovery/search — index the cleartext surface (paths, owners, branches, releases, reputation) for all repos; full content + symbol search for public repos (via the semantic graph). Private content is deliberately un-indexed server-side ("we can't read it, and that's the point"). The index is derived and disposable.
  • Storage path — FileBackend (today, fine for Lite) → S3/R2 with conditional writes (the CAS unlock, #68/#69) → multi-node with node-local rebuildable caches + etag invalidation.
  • Federation — already latent: a clone is a complete signed bundle, and contributions carry their own host DID and verify from the record alone, so instance B displays and verifies instance A's attestations without asking A. Define an instance↔instance mirror/subscribe + an attestation-trust list.

Trust model — and how a user catches the hub cheating

Cryptographically cannot: read private contents; forge a commit/veto/release/delegation/contribution (a forged record is kept and rendered unverified, never silently trusted); fabricate reputation (only host-attested merges count, and attestation is checkable); mint a delegation the owner didn't authorize. Can (the honest residual): withhold or lie by omission — serve a stale head, hide a commit, refuse to serve. It can't forge, but a mirror can censor. Build the counters as features:

  1. Re-clone and re-verify — a "Verify this repo" command that clones and reports any unverified record turns the trust model into a button.
  2. Head transparency — a signed append-only log of served (repo, view, head, time) (CT-style) makes equivocation detectable. This is exactly what issue #73 lays the groundwork for.
  3. Host attestation as the positive primitive — an attesting hub puts its own portable signature on "this merge happened here." ThadHub's brand becomes the value of its attestation — a manifesto-honest moat.

Monetization without betraying "your code & reputation are yours"

Never charge rent on the things the manifesto says are yours (leaving, verifying, taking your reputation). Charge for compute, hosting, availability, and the value of your attestation: (1) hosting/availability (free public, paid private storage/bandwidth/CDN); (2) CI/verification-as-a-service (sell the CPU-seconds that produce the signed check-provenance the land gate consumes — not merge permission); (3) paid higher-assurance attestations (your reputation to spend, not theirs to lose); (4) private-repo compute; (5) enterprise self-hosted ThadHub + federation. Off the table: charging to export, charging to read your own reputation, holding keys hostage.

The key insight: "ThadHub Lite" (H0) can ship this quarter over the already-live server with zero core changes, because public browse is "issue a world capability" and every metadata surface it renders is already cleartext in the pull bundle. The full federated, multi-node, private-browsing vision (H4–H6) is honestly gated on the S3/CAS + federation work the Thaddeus roadmap already sequences.

Hard open problems (state them plainly)

  • Browser key custody for private-repo decrypt — a key in localStorage is an XSS/phishing disaster. Likely answer: a browser extension / hardware signer exposing only sign/unseal to an allow-listed origin. Until then, private browse stays CLI/desktop-first and the web shows only the metadata skeleton. The single biggest UX gap vs. GitHub; scope it as its own project.
  • Discovery of encrypted content — you can't index what you can't read. Ship metadata search first; be honest that private content search is client-side.
  • Spam/Sybil at hub scale — free did:keys mean infinite identities; the defense is reputation that only counts host-attested merges (fresh DIDs have zero standing) + rate limits + reputation-weighted discovery. Depends on fixing the farming gap (#79).
  • Name squatting — handles are convenience, disputes resolve to DIDs; can't be "solved" without a central authority we refuse to be — so it's managed (reputation-gated/paid registration), not solved.
  • Moderation of a censorship-resistant substrate — the hub can't forge but can de-mirror; moderate its own mirror transparently (a public signed removal log) while being explicit that de-listing is not deletion. Needs real legal/policy design, not just engineering.
  • The multi-node correctness cliff — someone scaling ThadHub horizontally before the CAS work lands will corrupt head-repoints. Add a startup assertion that refuses multi-node until a CAS-capable backend is present.
